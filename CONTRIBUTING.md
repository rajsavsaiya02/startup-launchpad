# Contributing to Startup LaunchPad

We are excited that you are interested in contributing! A consistent process helps us maintain the quality of our codebase.

## Branching Strategy

-   All work must be done in a feature branch created from `develop`.
-   Branch names should be descriptive, using the format `feature/[feature-name]` (e.g., `feature/user-auth-api`) or `bugfix/[issue-name]`.

## Creating a Pull Request (PR)

1.  Ensure your branch is up-to-date with the latest changes from `develop`.
2.  Push your feature branch to the remote repository.
3.  Open a Pull Request from your branch to the `develop` branch.
4.  Fill out the PR template completely.
5.  At least one team member must review and approve the PR before it can be merged.

## Commit Message Convention

We follow the [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) specification. This helps us generate automated changelogs and makes the project history easy to read.

**Format:** `<type>: <description>`

-   **feat:** A new feature (e.g., `feat: implement user registration endpoint`)
-   **fix:** A bug fix (e.g., `fix: correct password validation logic`)
-   **docs:** Documentation changes (e.g., `docs: update setup instructions in README`)
-   **style:** Code style changes (e.g., `style: format code with Prettier`)
-   **refactor:** Code changes that neither fix a bug nor add a feature (e.g., `refactor: extract database connection logic`)
-   **test:** Adding or updating tests

Thank you for your contributions!
